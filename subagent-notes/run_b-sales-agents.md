## 一、总览结论

这 9 个仓库可以分成四类：

1. **销售对话代理框架**：`SalesGPT` 是最接近“AI 直接和客户对话并推进成交”的开源框架，核心是销售阶段感知、产品知识库检索、工具调用、付款链接/邮件/日程等动作闭环。
2. **销售情报与外呼前准备系统**：`ai-sales-team-claude` 不是客服式聊天机器人，而是面向销售人员的“并行研究团队”，把公司研究、联系人识别、BANT/MEDDIC 资格评估、竞争情报、外联策略拆成技能和子代理。
3. **销售教练 / 销冠能力复制产品**：`sales-ai-assistant`、`salemates`、`sales-ai-coach`、`Sales-Champion-Cloud-Group` 都围绕“把销冠经验结构化，再反哺一线销售”展开，但成熟度不同。它们共同强调录音/聊天记录分析、客户画像、质检评分、话术提炼、陪练、知识库和管理驾驶舱。
4. **早期原型 / 页面 / 占位仓库**：`SalesCrownAI`、`SalesChampionAI`、`ai-saler` 的可复用工程资产较有限，更多提供产品方向线索或 UI/后台模板。

对“AI 辅助销售工具”的最大启发是：不要只做一个会回答问题的聊天框。真正有销售推进价值的系统至少要覆盖四个闭环：

- **会前**：研究客户、识别决策链、判断是否值得投入。
- **会中 / 对话中**：识别销售阶段、提取需求与异议、给出下一步动作。
- **会后**：生成客户画像、质检报告、跟进计划、外联序列。
- **组织层**：沉淀销冠话术、形成团队评分标准、把高质量对话反哺知识库和培训。

## 二、逐仓库研究

## 1. filip-michalsky/SalesGPT

仓库地址：https://github.com/filip-michalsky/SalesGPT  
仓库描述：Context-aware AI Sales Agent to automate sales outreach.  
主要语言：HTML / Python 包形态，README 与 `pyproject.toml` 显示为 Python/LangChain 项目。  
开源热度：约 2600+ stars，属于本组中最有社区信号的项目。

### 1) 用意 / 目标

SalesGPT 的目标是构建一个**上下文感知的 AI 销售代理**，可以在语音、邮件、短信、WhatsApp、微信、Telegram 等渠道中自动进行销售沟通。

它的关键主张不是“生成一段话术”，而是让 AI 理解自己处于销售流程的哪个阶段，并据此采取合适行动。其默认销售阶段包括：

- 开场介绍
- 资格确认
- 价值主张
- 需求分析
- 方案呈现
- 异议处理
- 关闭 / 推进下一步
- 结束对话

代码中的 `salesgpt/stages.py` 明确定义了 8 个对话阶段；`salesgpt/agents.py` 中的 `SalesGPT` 控制器维护 `conversation_stage_id`、`current_conversation_stage`、`conversation_history`，并通过 `StageAnalyzerChain` 判断当前阶段。

### 2) 可复用能力

可复用能力较强，尤其适合作为“销售对话 agent”的核心参考：

- **销售阶段状态机**：把销售对话拆成有限阶段，并在每轮对话前后判断阶段迁移。
- **产品知识库检索**：`setup_knowledge_base` 使用文本切分、embedding、Chroma、RetrievalQA，将产品目录接入问答，降低幻觉。
- **工具调用式成交动作**：README 和 `tools.py` 展示了产品查询、Stripe 付款链接、邮件、Calendly 等工具方向。
- **多模型接入**：通过 LiteLLM 支持多种闭源/开源模型。
- **语音低延迟思路**：README 提到面向 voice channel 的低延迟目标，虽然需要结合具体部署验证。
- **人机协作与 tracing**：README 提到 human-in-the-loop 和 LangSmith tracing，适合生产化监控与人工接管。

### 3) 与销售推进的关系

SalesGPT 与销售推进关系最直接：它的设计目的就是让 AI 从开场到收款或预约下一步，参与完整销售流程。

它对推进的价值体现在：

- 不只“回答客户问题”，而是持续判断是否该进入资格确认、价值呈现、异议处理或 close。
- 客户问价格时可调用产品搜索，客户表达购买意图时可生成付款链接。
- 可把“下一步”设计成多种行动：付款、预约、邮件补充资料、转人工。

对实际销售工具而言，SalesGPT 的“阶段感知 + 工具动作”是最值得复用的骨架。

### 4) 对 AI 辅助销售工具的启发

- 销售 agent 需要有**显式阶段**，否则容易变成被动客服。
- 每个阶段都应对应不同的输出策略和工具权限。例如开场阶段不能过早报价，close 阶段需要明确下一步。
- 知识库检索必须和对话策略绑定：客户问事实问题时先查资料，客户表达情绪或异议时走销售方法论。
- 成交动作应可执行，而不只是建议。例如生成支付链接、预约链接、CRM 任务、跟进邮件。
- 可把“销售阶段判断”作为独立模型/链路，便于质检、回放和管理层分析。

### 5) 风险 / 不足

- 依赖 LangChain、Chroma、LiteLLM、OpenAI/Bedrock 等组件，生产部署复杂度不低。
- 默认示例偏通用 B2C/简单商品销售，复杂 B2B 多人决策链需要额外设计。
- 付款链接和邮件等工具调用存在越权、误售、错误报价、合规风险，必须加审批/置信度/规则围栏。
- 阶段判断若只靠 LLM，可能出现误判，例如客户只是试探价格却被推进到 close。
- 知识库以产品目录文本为基础，若缺少实时库存、价格、合同条款、禁用话术等约束，仍会有幻觉与承诺风险。

## 2. zubair-trabzada/ai-sales-team-claude

仓库地址：https://github.com/zubair-trabzada/ai-sales-team-claude  
仓库描述：AI-powered sales team for Claude Code. Research prospects, qualify leads (BANT + MEDDIC), find decision makers, generate outreach sequences, prepare for meetings, write proposals, and produce PDF pipeline reports — 14 skills, 5 parallel agents.  
主要语言：Python / Claude Code skill 体系。

### 1) 用意 / 目标

该项目目标是把 Claude Code 变成一支命令行里的 AI 销售团队。用户通过 `/sales prospect <url>` 等命令，触发完整的潜客研究、评分和销售材料生成。

README 展示的能力包括：

- 公司研究与 firmographics
- 决策人识别
- BANT + MEDDIC 线索资格评估
- 竞争情报
- 外联策略与邮件序列
- 会前准备
- 提案生成
- Markdown/PDF pipeline report

最核心的 `/sales prospect` 命令会启动 5 个并行代理：

- Company Research & Firmographics
- Decision Maker Identification
- Opportunity Assessment
- Competitive Intelligence
- Outreach Strategy & Messaging

### 2) 可复用能力

这是本组中“销售流程前半段”最有复用价值的项目。

可复用点包括：

- **技能化命令体系**：把销售任务拆成 `sales-prospect`、`sales-research`、`sales-qualify`、`sales-contacts`、`sales-outreach` 等技能。
- **并行子代理架构**：公司研究、联系人、机会评分、竞争情报、外联策略并行运行，最后汇总为 prospect score。
- **BANT/MEDDIC 评分模板**：`sales-qualify/SKILL.md` 对 Budget、Authority、Need、Timeline 的信号来源、打分规则、置信度进行了细化。
- **证据驱动研究流程**：明确要求使用网页、招聘、博客、定价页、案例、LinkedIn、新闻等公开信号，并区分 High/Medium/Low/Inferred confidence。
- **脚本化评分**：`scripts/lead_scorer.py` 实现 BANT 分数计算和 MEDDIC 完整度评估，可迁移成后端评分模块。
- **输出物标准化**：每个命令输出固定 Markdown 文件，例如 `PROSPECT-ANALYSIS.md`、`LEAD-QUALIFICATION.md`、`OUTREACH-SEQUENCE.md`。

### 3) 与销售推进的关系

它不直接替代销售与客户聊天，而是解决销售推进中的“是否值得追、该找谁、用什么理由切入、下一步做什么”。

销售推进价值体现在：

- 通过 prospect score 帮销售决定投入优先级。
- 通过 BANT/MEDDIC 判断线索质量，避免把时间浪费在低意向、低预算或无决策权对象上。
- 通过联系人和决策链研究，帮助销售从“找入口”进入“找经济买家/影响者”。
- 通过竞争情报和外联序列，提升首次触达质量。

如果 SalesGPT 是“对话执行者”，该项目更像“销售参谋部”。

### 4) 对 AI 辅助销售工具的启发

- 辅助销售工具应把“研究、评分、触达、准备、复盘”拆成可组合技能，而不是所有需求塞进一个聊天框。
- 线索评分必须保留证据和置信度，不能只给一个黑盒分数。
- BANT/MEDDIC、ICP、竞争情报可以成为 CRM 中每个客户的结构化字段。
- 对高价值客户，应支持多代理并行研究，以缩短销售准备时间。
- 输出物最好直接能行动：第一封邮件、会议提纲、异议预判、推荐下一步。

### 5) 风险 / 不足

- 依赖公开网页和搜索信号，容易受信息缺失、过期、抓取失败影响。
- 联系人识别可能涉及隐私、数据合规和准确性问题，尤其是使用 LinkedIn/社媒数据时。
- BANT/MEDDIC 分数在缺少一手销售 discovery call 的情况下只能作为“公开信号推断”，不能替代真实资格确认。
- 项目是 Claude Code skill 形态，直接作为 SaaS 产品复用需要重新设计权限、队列、UI、数据存储和审计。
- 容易诱导销售过度相信模型判断；必须明确 evidence、confidence、unknown。

## 3. hanxiao199001/sales-ai-assistant

仓库地址：https://github.com/hanxiao199001/sales-ai-assistant  
仓库描述：销冠AI分身 · AI房产销售赋能系统 | 录音进去，洞察出来。  
主要形态：Prompt 文档 + HTML 产品原型 + 落地计划。

### 1) 用意 / 目标

该项目面向房产中介团队，特别是学区房销售。核心目标是把老板/销冠的成交能力复制给团队。

README 总结为：

> 录音进去 → AI 分析 → 洞察出来

它试图解决三个痛点：

- 流量承接断层：老板引来的客户交给团队后信息损耗。
- 方向不统一：团队没有统一销售重点和跟进策略。
- 管理缺失：缺少自动生成的团队作战简报与质检机制。

### 2) 可复用能力

该仓库没有完整工程系统，但 prompt 和产品逻辑可复用性很高：

- **客户画像卡 Prompt**：从通话文字稿提取客户需求、预算、顾虑、决策链、意向度、跟进策略。
- **通话质检 Prompt**：基于 5 个维度打分：需求挖掘、信任建立、顾虑处理、学区专业度、跟进推进。
- **话术提炼 Prompt**：从老板成交录音中提炼场景化话术库。
- **房产垂直业务字段**：目标学区、房型、预算、入住/过户时间、购房目的、政策顾虑、家庭决策链。
- **产品原型**：HTML 原型覆盖流程图、客户画像卡、质检报告、团队作战简报等界面。
- **落地计划**：包括录音接入、ASR、客户画像、话术库、质检、合规检测、团队简报的阶段规划。

### 3) 与销售推进的关系

该项目的推进价值偏“会后复盘 + 下一步跟进”：

- 通过画像卡让接手销售快速知道客户状态，减少重复询问。
- 通过意向度评分和核心顾虑识别，指导后续跟进。
- 通过话术库复制老板处理异议的方式。
- 通过质检报告让销售知道哪里没有推进，例如没有约看房、没有挖决策人、没有回应政策顾虑。

它不强调 AI 直接替代销售对话，而是让销售团队在每通电话后更快形成作战动作。

### 4) 对 AI 辅助销售工具的启发

- “录音 → 转写 → 结构化画像 → 跟进建议”是非常清晰的低门槛 MVP。
- 垂直行业字段比通用 CRM 字段更重要。学区房场景中，“孩子入学时间”“政策顾虑”“家庭决策结构”就是销售推进关键。
- 质检评分不应只评语气，要评“是否推动了下一步”。
- 销冠复制不应保存逐字稿，而应沉淀“场景、策略框架、参考话术、禁忌说法”。
- 对团队管理来说，单通对话报告可以进一步汇总为每日/每周作战简报。

### 5) 风险 / 不足

- 当前是 Prompt + 原型阶段，没有真实 ASR、权限、CRM、自动上传、评分校准、数据闭环。
- 依赖人工复制文字稿给模型，难以规模化。
- Prompt 输出需要严格 schema 和校验，否则不同模型/不同输入会导致字段不稳定。
- 房产销售涉及政策、价格、房源、合规承诺，必须接入最新知识库并限制绝对化表述。
- 质检对“老板话术库”的依赖很强；如果老板录音样本不足或标注质量差，评分可能失真。

## 4. feng-crazy/salemates

仓库地址：https://github.com/feng-crazy/salemates  
仓库描述：SalesMate AI - 24/7 AI 销冠伙伴 (AI Sales Champion)。  
主要语言：Python。

### 1) 用意 / 目标

SaleMates 的愿景是构建部署在企业私域流量池中的 **24/7 自主销售智能体**，定位为“具备 Top Sales 思维模型的主动谈判专家与策略顾问”。

README 把终极形态描述为具有自我进化能力的销售合伙人，目标覆盖：

- 从破冰到成交的自动化
- 7x24 响应
- 智能转人工
- 通过历史数据优化话术
- 根据市场动态调整策略

### 2) 可复用能力

该项目在设计上很完整，且 README 声称多个模块已有实现。可复用方向包括：

- **意图识别与情绪分析**：IntentRecognizer + EmotionAnalyzer。
- **产品文档问答**：OpenViking 检索 + grounded 回复。
- **SPIN / FAB / BANT 销售方法论**：作为动态策略布局基础。
- **5 阶段销售状态机**：SalesStageStateMachine。
- **多渠道接入**：Feishu / Telegram 已部分接入。
- **安全围栏组件**：GuardrailManager、ConfidenceRouter、HumanHandoffManager。
- **销售方案生成**：ProposalGeneratorTool。
- **智能合同起草**：ContractDraftTool。
- **销售教练模式**：SalesCoachTool，包含实时陪练、实时助攻、评分反馈。
- **长期记忆与技能市场**：MemoryStore、EnhancedMemoryManager、workspace/skills。
- **Docker 部署**：docker-compose 配置。

### 3) 与销售推进的关系

SaleMates 把销售推进看作一个 OODA 循环：

- Observe：识别客户意图、情绪、历史画像、知识库事实。
- Orient：判断销售阶段、选择 SPIN/FAB/BANT 等策略。
- Decide：决定回复、提问、转人工、生成方案或合同。
- Act：在私域渠道里继续对话或输出销售材料。

它的推进价值较全面：不仅能在对话中推进，还把方案、合同、陪练、安全路由都纳入销售闭环。

### 4) 对 AI 辅助销售工具的启发

- 私域销售 agent 必须有**安全围栏和转人工机制**，尤其涉及价格、合同、法务、情绪冲突时。
- 销售方法论可以作为策略选择器，而不是固定话术树。
- 客户画像应随对话持续更新，而不是只在首次接触后生成一次。
- “方案生成”和“合同草拟”是从对话走向交易的关键动作，但必须带审批。
- 教练模式和自动销售模式可以共用底层能力：同一套评分、方法论和知识库，既可辅助人，也可驱动 agent。

### 5) 风险 / 不足

- README 中也承认部分关键能力未实现或未集成，例如客户画像提取、千人千面个性化、主动跟进引擎、安全组件尚未集成到 AgentLoop。
- 愿景很大，容易出现模块多但闭环不深的问题。
- 合同和报价等能力风险较高，必须有权限、审批、审计、版本化模板和法务规则。
- 情绪识别、微情绪分析若缺少多模态数据和验证集，容易被过度包装。
- 自我进化、自动 A/B 测试需要严谨实验设计，否则可能优化到短期转化而伤害品牌或合规。

## 四、对 AI 辅助销售工具的关键启发

## 1. 产品架构应拆成“代理、教练、参谋、管理后台”四层

单一聊天框很难覆盖销售全流程。更稳妥的产品架构是：

- **AI 销售代理**：面向客户，负责对话、问答、推进下一步。
- **AI 销售教练**：面向销售，负责陪练、话术建议、复盘评分。
- **AI 销售参谋**：面向销售/BD，负责线索研究、BANT/MEDDIC、联系人、竞争情报、外联序列。
- **老板/管理后台**：面向管理者，负责排行、质检、知识沉淀、高频异议、团队能力地图。

SalesGPT 强在代理，ai-sales-team-claude 强在参谋，sales-ai-coach 强在教练，Sales-Champion-Cloud-Group 强在管理后台和行业闭环。

## 2. 销售推进必须显式建模

可复用的推进字段包括：

- 当前销售阶段
- 客户预算
- 决策人 / 影响人
- 需求 / 痛点
- 时间线
- 顾虑 / 异议
- 竞品或替代方案
- 下一步动作
- 跟进截止时间
- 置信度
- 证据来源

这些字段应贯穿对话 agent、教练复盘、CRM 和管理后台，而不是只存在于一次性模型输出里。

## 3. BANT/MEDDIC 与销冠话术库可以结合

英文 B2B 项目常用 BANT/MEDDIC，中文销冠项目常强调“老板话术库/销冠经验”。两者可以融合：

- 用 BANT/MEDDIC 做线索资格和销售阶段判断。
- 用销冠话术库做具体异议处理和话术推荐。
- 用真实成交结果反向校准评分规则和推荐策略。

## 4. 低风险 MVP 应从“录音/聊天记录复盘”切入

相比让 AI 直接和客户成交，先做“录音/聊天记录 → 客户画像 → 质检 → 跟进建议”风险更低，也更容易证明价值。

最小可行闭环可以是：

1. 上传通话录音或聊天截图。
2. ASR/OCR 转写。
3. 抽取客户画像、BANT、异议、下一步。
4. 生成销售质检评分。
5. 推荐跟进话术和 CRM task。
6. 管理后台汇总团队问题和高频异议。

这一路径可以借鉴 `sales-ai-assistant` 和 `sales-ai-coach`。

## 5. 高阶能力是“从建议到动作”

真正提高转化的不只是生成建议，而是创建可执行动作：

- 写好第一封邮件 / 跟进邮件。
- 生成会议准备 brief。
- 创建 CRM 跟进任务。
- 生成报价/方案草稿。
- 生成预约链接。
- 在安全条件满足时生成付款链接。
- 在风险触发时转人工。

SalesGPT 和 SaleMates 都指向这条路，但也带来更高安全风险。

## 6. 证据链和置信度是生产化底座

多个项目都暗示或明确需要 grounded 输出：

- SalesGPT 通过产品知识库减少幻觉。
- SaleMates 强调 OpenViking 检索与不确定时转人工。
- Sales-Champion-Cloud-Group 强调 evidence sources。
- ai-sales-team-claude 明确区分 High/Medium/Low/Inferred confidence。

建议任何销售 AI 输出都包含：

- 结论
- 依据
- 置信度
- 不确定项
- 推荐下一步
- 是否需要人工确认

## 五、风险与不足清单

## 1. 销售合规风险

AI 可能错误承诺价格、优惠、库存、交付周期、学区政策、合同条款、医疗/金融收益等。涉及高风险行业时，必须有禁用话术、强制引用、审批和转人工。

## 2. 幻觉与过度推断

公开网页研究、客户画像、意向度评分都可能基于不完整信息。系统应把“未提及”和“推断”分开，避免把推断写成事实。

## 3. 数据隐私风险

录音、聊天截图、客户联系方式、预算、家庭情况、企业联系人都属于敏感销售数据。需要最小化采集、脱敏、权限控制、审计日志和数据删除机制。

## 4. 评分失真风险

销售质检和销冠指数如果没有真实成交结果、客户满意度和长期复购校准，容易奖励“话术完整”而非“真实有效”。

## 5. 自动化越权风险

付款链接、合同起草、报价、优惠承诺、外联邮件自动发送都应分权限。建议默认是草稿，只有低风险场景才自动执行。

## 6. 行业迁移风险

房产、茶叶、SaaS、医疗、金融、电商的销售推进字段完全不同。通用框架可复用，但行业知识库、评分维度、禁用话术、成交动作必须重建。

## 7. 工程复杂度风险

多模型、OCR、ASR、RAG、CRM、渠道接入、权限、后台、评估集会迅速增加复杂度。应先选择一个闭环做深，而不是一次性做“全自主销售合伙人”。

## 六、建议的组合式产品方案

如果要基于这些仓库的启发设计一个可落地的 AI 辅助销售工具，建议按以下路线组合：

### Phase 1：销售复盘 MVP

借鉴：

- `sales-ai-assistant`
- `sales-ai-coach`
- `Sales-Champion-Cloud-Group`

能力：

- 上传录音/聊天截图。
- ASR/OCR 转写。
- 生成客户画像、BANT、核心异议、下一步。
- 生成质检报告和改进建议。
- 管理后台统计高频问题、团队短板、销冠话术。

目标：低风险证明“节省复盘时间 + 提升跟进质量”。

### Phase 2：销售参谋与外联

借鉴：

- `ai-sales-team-claude`

能力：

- 公司研究。
- 决策链识别。
- BANT/MEDDIC 评分。
- 竞争情报。
- 外联邮件序列。
- 会议准备 brief。

目标：提升获客和会前准备质量。

### Phase 3：实时销售助手

借鉴：

- `SalesGPT`
- `salemates`
- `sales-ai-coach`

能力：

- 实时阶段识别。
- 客户异议识别。
- 销售话术建议。
- 知识库引用。
- 下一步动作推荐。
- 人工确认后生成邮件/任务/方案。

目标：在销售进行中提供助攻，而不是一开始就替代销售。

### Phase 4：受控自动销售代理

借鉴：

- `SalesGPT`
- `salemates`

能力：

- 面向低风险、标准化商品/服务的自动对话。
- 工具调用：查知识库、查产品、预约、发资料、生成草稿。
- 置信度路由、情绪熔断、价格/合同红线、转人工。

目标：只在边界清晰的场景中自动化成交动作。

## 七、最值得重点跟进的仓库

优先级建议：

1. **SalesGPT**：学习销售阶段感知与工具调用式成交。
2. **ai-sales-team-claude**：学习销售研究、BANT/MEDDIC、并行子代理和证据评分。
3. **sales-ai-coach**：学习销售教练平台、陪练、OCR、复盘和 Agent Harness。
4. **Sales-Champion-Cloud-Group**：学习中文行业化销售运营 MVP、老板驾驶舱、销冠指数、证据链。
5. **sales-ai-assistant**：学习垂直行业 prompt、录音复盘和销冠话术提炼。
6. **salemates**：学习私域销售 agent 的宏观设计、安全围栏和方案/合同/教练组合。

低优先级：

- `ai-saler`：可参考管理后台方向，但 AI 核心能力不足。
- `SalesCrownAI`：可参考官网包装，不适合作为能力实现参考。
- `SalesChampionAI`：信息过少，暂不建议投入研究。

## 八、最终判断

这批仓库说明 AI 销售工具正在从“话术生成器”走向“销售操作系统”。最有价值的不是某一个模型 prompt，而是把销售推进拆成可观测、可评分、可复盘、可执行的工作流：

- 对客户：AI 能更快理解需求、回答事实问题、推动下一步。
- 对销售：AI 能复盘对话、推荐话术、陪练异议处理。
- 对管理者：AI 能量化团队能力、沉淀销冠经验、发现高频问题。
- 对系统：AI 必须有证据链、置信度、安全围栏和人工接管。

因此，若要建设自己的 AI 辅助销售产品，推荐优先做“销售复盘 + 画像 + 质检 + 跟进建议”的低风险闭环，再逐步叠加销售研究、实时助攻和受控自动化成交。
